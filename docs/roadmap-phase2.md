# Phase 2 路线图：从工具到系统

> Phase 1 完成了归因→风控→信息压缩的离线闭环，Phase 2 目标是**自动化 + 实时化**，让系统每天自己跑起来。

## 总览

```
Phase 1（已完成）                    Phase 2（规划中）
┌──────────────────┐               ┌──────────────────────────────────┐
│ 手动跑脚本        │               │ 自动触发 + 主动推送               │
│ PDF 导入数据      │    ──→        │ 券商 API 实时持仓                │
│ 本地看报告        │               │ 微信收简报 + 告警                 │
│ 信息源零散        │               │ 可信信息源体系                    │
└──────────────────┘               └──────────────────────────────────┘
```

---

## P1：每日复盘助手（券商 API 驱动）

**目标**：告别手动导 PDF，通过券商接口自动获取持仓 & 成交，每天收盘后自动完成复盘。

**现状 → 目标**：
- 现在：PDF 对账单 → `convert_broker_data.py` → trades.csv → 手动跑归因/风控
- 目标：券商 API → 自动拉持仓/成交 → 自动跑归因+风控+复盘 → 推送结果

**关键设计**：
- [ ] 券商接口适配层（`shared/broker_api.py`）
  - 统一接口：`get_holdings()` / `get_trades(date)` / `get_account_summary()`
  - 首选：东方证券 API（与现有 PDF 数据源一致）
  - 备选：富途 OpenD（已有配置，港股也能覆盖）
- [ ] 自定义指标插件机制
  - 复盘模板支持注入自定义指标（如：偏离均线幅度、板块轮动信号）
  - 指标以 Python 函数注册，复盘 prompt 自动拼接
- [ ] 数据管道：券商 API → 标准 CSV（兼容现有 trades.csv / holdings.csv 格式）
  - 增量更新，不破坏历史数据

**依赖**：券商 API 开通 & 鉴权

---

## P2：自动调度 + 微信推送（腾讯云 qclaw）

**目标**：每天定时自动触发全流程，结果推送到微信，不用开电脑也能收到复盘。

**架构**：
```
腾讯云 qclaw（定时触发）
    │
    ├─ 15:30  收盘后 → 拉持仓 → 风控检查 → 复盘 → 推送微信
    ├─ 08:00  盘前   → 持仓情报摘要 → 推送微信
    └─ 不定时  → 重要公告检测 → 告警推送
    │
    ▼
微信推送（企业微信机器人 / Server酱 / PushPlus）
```

**关键设计**：
- [ ] 云函数入口（`deploy/qclaw_handler.py`）
  - 封装现有 `quickstart.sh` 的 `risk` / `brief` / `review` 流程
  - 环境变量管理：券商 token、LLM API key、推送 token
- [ ] 微信推送适配（`shared/notifier.py`）
  - 支持多通道：企业微信 Webhook / Server酱 / PushPlus
  - Markdown 格式推送，关键数据高亮
  - 告警分级：普通信息 / 风险预警 / 紧急告警
- [ ] 部署脚本 & 配置
  - `deploy/` 目录：qclaw 配置、依赖打包、环境变量模板

**依赖**：P1（券商 API）完成后才能全自动；推送通道可先行开发

## P3：每日信息日报（可信信息源体系）

**目标**：构建自己的信息源矩阵，替代刷新闻的低效方式，LLM 压缩后推送。

**现状 → 目标**：
- 现在：信息摘要仍依赖单一快讯源（噪音大）
- 目标：多源聚合 → 去重去噪 → LLM 压缩 → 结构化日报

**信息源矩阵（逐步构建）**：

| 优先级 | 信息源 | 内容 | 接入方式 |
|--------|--------|------|---------|
| P0 | 交易所公告 | 持仓股公告（财报/分红/增减持） | 巨潮资讯 API / 爬虫 |
| P0 | 东方财富快讯 | 市场要闻 | 已有 `get_eastmoney_news()` |
| P1 | 财联社电报 | 实时快讯 | RSS / API |
| P1 | 央行/统计局 | 货币政策、经济数据 | 官网 RSS |
| P2 | 雪球/同花顺 | 个股讨论热度 | 爬虫（注意合规） |
| P2 | 研报摘要 | 行业/个股研报 | 东方财富研报中心 |

**关键设计**：
- [ ] 信息源注册框架（`shared/news_sources/`）
  - 每个源一个 adapter，统一输出 `NewsItem(title, content, source, time, relevance)`
  - 可信度评分：官方公告 > 主流财经 > 社区讨论
- [ ] 去重 & 关联
  - 同一事件多源报道去重
  - 与持仓股自动关联，标注相关度
- [ ] LLM 压缩管道升级
  - 分层摘要：先按源压缩，再跨源综合
  - 持仓相关信息优先、加粗标注

**依赖**：可独立开发，逐步接入信息源

## P4：持仓重要事件提醒

**目标**：财报日期、分红除权、重大公告等关键事件，提前提醒 + 当天推送。

**覆盖事件**：

| 事件类型 | 提醒时机 | 推送内容 |
|---------|---------|---------|
| 财报发布 | 发布前 3 天 + 当天 | 预期 vs 实际，历史对比 |
| 分红除权 | 公告日 + 除权前 1 天 | 分红方案、除权价、是否需要操作 |
| 大股东增减持 | 公告当天 | 增减持比例、历史模式 |
| 重大关联交易 | 公告当天 | 交易金额、对业绩影响评估 |
| 股权质押变动 | 公告当天 | 质押比例变化、风险评估 |
| 限售股解禁 | 解禁前 5 天 | 解禁规模、历史解禁后走势 |

**关键设计**：
- [ ] 事件日历（`shared/event_calendar.py`）
  - 数据源：巨潮资讯 / 东方财富事件日历 API
  - 每日更新持仓股的未来事件
  - 本地缓存 + 增量更新
- [ ] 提醒引擎（`shared/event_alerter.py`）
  - 规则配置：哪些事件提前几天提醒
  - 与 P2 的推送通道复用
- [ ] LLM 增强
  - 公告原文 → LLM 提取关键信息 + 影响评估
  - 结合持仓成本/仓位给出操作建议参考

**依赖**：P2（推送通道）、P3（巨潮资讯源可复用）

## 实施节奏

```
2026年4月                 5月                  6月                  7月
  │                       │                    │                    │
  ├─ P1: 券商API适配 ─────┤                    │                    │
  │   调研+开发            │                    │                    │
  │                       ├─ P2: qclaw+推送 ───┤                    │
  │                       │   部署+联调         │                    │
  ├─ P3: 信息源(持续) ────────────────────────────────────────────── │
  │   逐步接入             │                    │                    │
  │                       │                    ├─ P4: 事件提醒 ─────┤
  │                       │                    │   日历+告警         │
  │                       │                    │                    │
  ▼                       ▼                    ▼                    ▼
  券商数据跑通             全自动+微信收报告     信息源≥3个           事件提醒上线
```

**原则**：
- P1 是基础，券商 API 跑通后其他模块才能全自动
- P2 推送通道可以先用现有数据源验证，不必等 P1
- P3 是长期工程，边用边加源，不追求一步到位
- P4 依赖 P2 + P3 的基础设施，放最后
